NetPlay

NetPlay lets two devices play the same game together over Wi-Fi/network, one acting as Host and the other as Client. Both devices run the exact same ROM at the same time; only controller inputs travel over the network.

Modes

Only the Host chooses the mode, from the NetPlay dialog before creating the game. The Client doesn't need to pick anything: it automatically adopts whatever mode the Host is using as soon as it connects.

  • Lockstep: the safest, most compatible mode. Both devices wait for each other's input before advancing a frame, so gameplay is always perfectly synchronized. The trade-off is input lag: the worse the connection (higher ping), the more delay you'll feel between pressing a button and seeing it happen. Works with every game.
  • Rollback: the low-latency mode, ideal for fast-reflex games (NeoGeo, CPS1/CPS2, fighting games...). Each device predicts the peer's input and keeps playing immediately; if the prediction turns out wrong once the real input arrives, it silently rewinds a few frames and re-simulates them with the correct input. Most of the time you don't feel any lag at all. Costs more CPU/battery than Lockstep, and needs extra RAM to keep a short history of savestates ready to rewind to.

Input Delay

Both modes hide network latency by delaying local input a few frames before it takes effect (so it arrives at the same time as the matching peer input), instead of applying it instantly. By default this is set to Auto: MAME4droid measures your connection's ping and adjusts the delay by itself, so you don't need to touch anything. It works a little differently depending on the mode:

  • In Rollback, Auto continuously tracks about half the round-trip time, within safe bounds — kept low, since rollback's own prediction/rewind absorbs whatever latency the delay doesn't cover.
  • In Lockstep, Auto adjusts in occasional steps (scheduled to land on the same frame for both devices), reacting to sustained changes in latency rather than every small ping fluctuation.

If you'd rather not leave it on Auto, Options->Settings->NetPlay->NetPlay delay lets you pin it to a fixed value (1 to 6 frames) instead. As with the mode, this is effectively decided by the Host: once connected, the Client adopts whatever delay the Host ends up using for the session.

Rollback compatibility and automatic fallback

Rollback needs to save/restore the full machine state many times per second, so it's not equally suited to every game. MAME4droid checks this automatically at the start of a Rollback session and falls back to Lockstep (with an on-screen warning) when needed — you don't have to do anything:

  • Savestate too large: games with a lot of RAM (e.g. those using a hard disk/CHD) produce a savestate too big to keep many copies of in memory. If it doesn't fit the configured RAM budget, that session automatically runs in Lockstep instead.
  • Different savestate layout across devices: the two devices must register the exact same internal save data for the driver (same MAME4droid build/version). If a mismatch is detected before the game state is exchanged, both peers fall back to Lockstep rather than risk a silent desync.

Games known to have trouble with Rollback

A handful of games read an analog control (steering wheel, spinner / paddle, pedals) through a secondary chip (a protection MCU or a second sound CPU) that polls it at hardware-timing precision instead of once per frame. Rollback's rewind-and-replay can then capture a slightly different analog reading than the original run did, which can show up as an occasional desync in these games:

  • Arkanoid (and clones/bootlegs sharing its paddle + protection MCU hardware)
  • Buggy Boy / Buggy Boy Jr. / TX-1 family (steering wheel + pedals read by a second sound CPU)

If you notice desyncs on a game like this, choose Lockstep for it instead — Lockstep never rewinds or re-simulates, so this class of issue can't happen there. Other games with similar hardware (analog dial/paddle/wheel/pedal controls read outside the normal per-frame input path) could show the same behavior.

Connection stats overlay

While a session is active, a small ping/delay indicator can optionally show at the bottom of the screen: current mode, ping (round-trip time) and input delay, colored green for a good connection, yellow for a noticeable one and red for a poor one. It refreshes every couple of seconds and disappears automatically once the session ends. Turn it on under Options->Settings->NetPlay->Show connection stats overlay (off by default).

Desync detector

By default, a Rollback session also runs a background check that compares a periodic state checksum between both devices. If it ever finds the two screens have genuinely drifted apart, it shows a short red warning telling you the current game may not be Rollback-safe — that's your cue to try Resync or switch that session to Lockstep. The check costs a little extra CPU every frame; if you'd rather save that cost on an older device and are fine giving up the warning, turn it off under Options->Settings->NetPlay->Desync detector.

Hosting a game

  • Open the NetPlay dialog and tap Start Game.
  • Pick Lockstep or Rollback for this session.
  • MAME4droid opens a listening port (configurable under Options->Settings->NetPlay Port, default 2080) and the waiting screen shows your addresses, each labeled with its use: Your local IP for a player on the same Wi-Fi / network, Your public IP for a player joining over the internet — the Share button sends them through any messenger, ready to paste into the other player's Join box.
  • Load/select the game you want to play; once the Client connects, it will be launched automatically on both devices.

Joining a game

  • Open the NetPlay dialog and tap Join Game.
  • Enter the Host's IP address (and make sure you're both using the same port).
  • That's it — no mode to choose. As soon as the connection is established, your device adopts the Host's mode and loads the Host's selected game automatically.

Note: both devices need to be reachable on the same network (same Wi-Fi, or otherwise routed so the chosen port is open between them), need the exact same ROM file, and should be running the same MAME4droid version — a version mismatch is detected and the connection is refused with a warning instead of risking a silent desync.

Internet play (no server needed)

Two players on different networks can connect directly by exchanging addresses over any messenger — no account, lobby or server involved. Hosting works exactly the same as on a LAN: tap Start Game and read the waiting screen — it tells you what to share and what to do next.

The usual case is automatic (UPnP): while you host, MAME4droid asks your router to open the NetPlay port by itself. If the waiting screen says Router port mapped (UPnP), just tap Share and send your addresses to the other player: they tap Join Game and paste the internet one. That's all — it works even if the other player is on mobile data. The mapping is removed when the session ends; the feature can be turned off under Options->Settings->NetPlay->UPnP port mapping.

If UPnP is not available (the line doesn't appear, or the join won't land), MAME4droid can still hole-punch through both routers. Punching needs the addresses exchanged BOTH ways:

  • Step 1 (Host): tap Share on the waiting screen and send your addresses to the other player.
  • Step 2 (Client): Join the Host's public address; then tap Share on the connecting screen to send your own public address back to the Host.
  • Step 3 (Host): tap Peer IP on the waiting screen and paste the Client's address. The connection usually lands within a few seconds.

Requirements and limitations:

  • Works between most home Wi-Fi routers. Over mobile data (4G/5G) hole punching usually fails: carrier CGNAT tends to change the public port per connection — even when no Symmetric warning was shown. A mobile-data player can still join a Host whose port is forwarded (UPnP above, or the manual fallback below).
  • If Symmetric NAT detected appears, internet play may fail — try anyway, or use the port-forwarding fallback.
  • Strict or gaming routers (SPI firewall — e.g. ASUS TUF/ROG) may silently drop punched connections even when both NATs look fine. If the router settings offer a NAT type option, choose Full Cone; otherwise use the port-forwarding fallback.
  • If both players are behind CGNAT (no reachable public IPv4 at either end), neither hole punching nor port forwarding can work. That is an ISP limitation, not an app bug; ask your ISP for a public IPv4 or play on a network that has one.
  • Expect a higher ping than on a LAN — Rollback mode is strongly recommended.
  • If both devices are on the same Wi-Fi, use the normal local flow (local IP), not the public addresses.

Fallback that always works: the Host forwards UDP port 2080 to their device on their router (routers call this Virtual Server, Port Forwarding, or a custom Open NAT game profile) and shares their public IP; the Client simply Joins it. Only the Host's router needs this — it also lets a Client on mobile data play. It is the manual equivalent of UPnP, for routers where UPnP is off or broken.

IPv6 (Network protocol)

Options->Settings->NetPlay->Network protocol selects how the devices connect: IPv4 (default), IPv6 or Auto (IPv6 + IPv4). IPv6 has no NAT: each device gets its own public address, so internet play usually connects directly — no port forwarding, no UPnP, and the CGNAT limitation above disappears. In particular, two phones on mobile data (4G/5G) with IPv6 — which most carriers provide — connect to each other directly with zero router setup: this is the easiest and most reliable way to play over the internet. The default is IPv4 because it is the most field-tested path, but Auto is worth trying: it works with every peer and picks the best route by itself.

  • IPv4: the classic behavior described above.
  • IPv6: both players connect over IPv6 only. The shared invite carries a [bracketed] address; Share and Paste work exactly the same.
  • Auto: the Host accepts both families and shares every address it has (IPv6, IPv4 and local network); the Client automatically picks whichever works with its own connection. Choose this if you're not sure what the other player has.

Both players should use the same setting (Auto pairs fine with either). Some routers ship an IPv6 firewall that blocks inbound connections: the built-in hole punching normally opens it by itself, and otherwise allowing the NetPlay port in that firewall is the IPv6 equivalent of port forwarding.

If your router has IPv6 enabled but your ISP hands out no public IPv6 prefix, your devices still get a same-network (ULA) address: internet play isn't possible, but two devices on the same network can still play over IPv6. The Host lists that address labelled same network only.

During the game

  • Resync: on a live Rollback session, the NetPlay dialog offers a Resync button. It forces both devices to freeze for a moment while the Host re-sends its exact game state to the Client, snapping both machines back into sync. Use it if you ever notice the two screens have visibly drifted apart. Not needed (and not shown) in Lockstep, since Lockstep can't drift in the first place.
  • Disconnect: ends the NetPlay session for both devices and returns to normal offline play.

Tips

  • A stable Wi-Fi connection matters much more than raw speed — a stable 2.4GHz link often works better than a congested 5GHz one.
  • Keep both devices' screens on and the app in the foreground during the session; backgrounding can pause the connection.
  • If Rollback feels choppy on a particular game, try switching that session to Lockstep — some hardware (see above) simply isn't a good fit for rollback.
